# 关于Commit规范

# 目的

  • 加快Reviewing Code的过程
  • 帮助我们写好Release Note (Changelog)
  • 快速追踪分支及commit
  • 方便确认代码提交者及commit（git blame）
  • 我们可以通过commit来回溯项目
    • 一次优秀的代码提交应该包含什么?

# Angluar团队的提交代码方式

提交格式：

<类型>[可选的作用域]: <描述>

[可选的正文]

[可选的脚注]

type类型：

  • feat 新功能(feature)
  • fix 修补bug
  • docs 文档
  • style 格式调整（不影响代码功能）
  • refactor 重构（代码优化，不影响功能）
  • test 测试
  • chore 杂项 （构建工具、辅助工具的变动）
  • perf 性能优化

示例：git commit -m 'fix(#1234): 修复XXX'

# 一次优秀的代码提交应该包含什么

  • 每次提交只做一个修改
  • 每次提交要包含完整修改
  • 写好注释，说明你修改了什么
  • 注释说明为什么做这个修改
  • 不要提交被注释掉的代码
